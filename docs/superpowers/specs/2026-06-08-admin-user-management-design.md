# Управление пользователями в админке — Design

**Дата:** 2026-06-08
**Статус:** draft
**Скоуп:** итерация 1 из 2 (рассылки — следующий спек)

---

## 1. Цель и контекст

В текущей админке (`admin/`) есть управление контентом (объекты, новостройки, застройщики) и заявками, но **нет интерфейса для работы с пользователями**. Администратор не может:

- посмотреть список зарегистрированных пользователей;
- заблокировать нарушителя (`is_active=0`);
- подтвердить email вручную, если письмо не дошло;
- создать пользователя (агента или застройщика) самостоятельно;
- отправить пользователю ссылку на сброс пароля.

Результат — невозможность реагировать на инциденты и ручное создание учёток через SQL/phpMyAdmin.

**Не входит в этот спек** (явно отложено):
- Audit log действий администратора — потребуется позже при рассылках.
- Bulk actions и экспорт CSV — пригодится для сегментов в рассылках.
- Email-уведомления пользователю о бане/подтверждении — отдельная фича.
- 2FA — снят с повестки ранее по решению пользователя.
- Рассылки — это итерация 2 со своим спеком.

## 2. Роли пользователей

В `users.role` ENUM — `admin` / `agent` / `user`. **Не вводим новых ролей.**

| Роль | Описание | Может создавать админ? |
|------|----------|------------------------|
| `admin` | Полный доступ ко всей админке | ❌ Нет (защита от потери единственного админа — при создании выбора роли admin не предлагаем) |
| `agent` | Агент по недвижимости (в перспективе — свои объекты, ответы на заявки) | ✅ Да |
| `user` | Обычный покупатель/арендатор | ✅ Да |

Существующего единственного админа понизить или заблокировать через админку **нельзя** (защита «последнего админа»).

## 3. Действия, которые админ может выполнить

| Действие | Где в UI | Серверный эндпоинт | Эффект в БД |
|----------|----------|--------------------|-------------|
| Просмотр списка | `admin/users.php` | GET | — |
| Поиск/фильтрация списка | `admin/users.php` | GET `?q=…&role=…&status=…` | — |
| Создание пользователя | `admin/user-edit.php` (без `?id=`) | POST | `INSERT INTO users` + email с паролем |
| Редактирование пользователя | `admin/user-edit.php?id=N` | POST | `UPDATE users` SET … |
| Бан / разбан | inline-кнопка в `admin/users.php` | POST `?action=toggle-active&id=N` | `is_active = NOT is_active` + сброс `failed_login_attempts`, `locked_until` |
| Подтверждение email | inline-кнопка в `admin/users.php` | POST `?action=verify-email&id=N` | `email_verified_at = NOW()` (только если NULL) |
| Отправка ссылки на сброс пароля | `admin/user-edit.php?id=N` | POST `?action=send-reset&id=N` | `INSERT INTO password_resets` + email |
| Удаление пользователя | `admin/user-edit.php?id=N` | POST `?action=delete&id=N` | `DELETE FROM users` (с каскадом по `favorites`/`inquiries`/`reviews`) |

**Сброс пароля «с установкой нового пароля»** — **отсутствует в этой итерации**. Причина: админ не должен знать пароли пользователей. Если пользователю нужен сброс — он делает это сам через `forgot-password.php`, либо админ отправляет ему ссылку на сброс (см. таблицу выше).

## 4. Архитектура

- **Файлы:**
  - `admin/users.php` — список + inline-действия (бан, подтверждение email, отправка ссылки на сброс) + поиск/фильтры/пагинация.
  - `admin/user-edit.php` — создание (`?id` нет) или редактирование (`?id=N`).
- **Доступ:** `requireAdmin()` (уже используется во всей админке).
- **Стек:** vanilla PHP + PDO + существующий `css/admin.css`. Без JS-фреймворков. Минимум vanilla-JS для `confirm()` на опасных действиях.
- **БД:** **0 миграций.** Используем существующую `users` (поля `is_active`, `email_verified_at`, `failed_login_attempts`, `locked_until`, `role`, `oauth_provider`, `oauth_id`) и `password_resets` (уже есть из миграции 020).
- **CSRF:** `csrf_token()` / `verifyCsrfToken()` (используем существующие helper'ы из `includes/auth/`).
- **Flash-сообщения:** `$_SESSION['flash']` — короткий текст + тип (`success` / `error`), показывается в верхней страницы, очищается после показа.
- **Никаких новых таблиц в этой итерации.**

## 5. UI: страница списка `admin/users.php`

### Layout

- Стандартный `admin-header` + `admin-sidebar` (с новым пунктом «Пользователи», см. §10).
- `<main class="admin-main">` с заголовком «Пользователи» и счётчиком «N всего».

### Тулбар

```
┌────────────────────────────────────────────────────────────────────┐
│ Пользователи (всего 247)                  [ + Создать ]           │
├────────────────────────────────────────────────────────────────────┤
│ [Поиск: email / имя / телефон ____________] [🔍]                   │
│ Роль: [Все ▾]   Статус: [Все ▾]   Email: [Все ▾]   [ Сбросить ]   │
└────────────────────────────────────────────────────────────────────┘
```

- **Поиск** — `<input name="q">`, LIKE по `email` / `first_name` / `last_name` / `phone`. Submit по Enter; опционально — 300 мс debounce на JS-submit.
- **Фильтры:**
  - `role` ∈ `all` / `admin` / `agent` / `user`
  - `status` ∈ `all` / `active` (is_active=1) / `blocked` (is_active=0) / `verified` (email_verified_at IS NOT NULL) / `unverified` (email_verified_at IS NULL)
- Фильтры — обычные GET-параметры. При изменении `<select>` — автосабмит формы (JS).
- Кнопка «Сбросить» возвращает на `?` без параметров.

### Таблица

```
ID │ Email                │ ФИО             │ Роль  │ Статус      │ Email      │ Зарегистрирован │ Действия
───┼──────────────────────┼─────────────────┼───────┼─────────────┼────────────┼──────────────────┼─────────
42 │ petrov@mail.ru       │ Иванов Пётр     │ agent │ ✓ Активен   │ ✓ Подтв.   │ 12.04.2026       │ ✎ 🔄 ✓✉ 🔒
43 │ spam@example.com     │ —               │ user  │ ✗ Заблок.  │ ⏳ Не подт.│ 14.04.2026       │ ✎ 🔄 ✓✉ 🔓
1  │ admin@elsesser…      │ Elsesser A.     │ admin │ ✓ Активен   │ ✓ Подтв.   │ 01.01.2025       │ ✎        ← только редактирование, без бан-кнопок
```

- **Пагинация:** 25 строк/стр., GET-параметр `?page=`. Используем существующий паттерн из `properties.php` (или `admin/properties.php` — посмотреть при планировании).
- **Сортировка:** по умолчанию `ORDER BY created_at DESC`. Дополнительная сортировка по столбцам — **отложена**.
- **Действия** (одна строка, иконки + tooltip):

| Иконка | Действие | Когда скрыта |
|--------|----------|--------------|
| ✎ (fa-pen) | Редактировать | Никогда |
| 🔄 (fa-sync) | Бан / разбан | Если это сам админ (`$_SESSION['user_id']`) |
| ✓✉ (fa-envelope-open-text) | Подтвердить email | Если `email_verified_at IS NOT NULL` (уже подтверждён) |
| 🔒 / 🔓 | — | См. выше (та же кнопка, что и бан) |

Все действия — POST с CSRF-токеном. Кнопка «Заблокировать» — `confirm("Заблокировать пользователя …?")` перед отправкой.

### Защита «последнего админа»

Перед рендером списка считаем:
```php
$adminCount = (int)$pdo->query("SELECT COUNT(*) FROM users WHERE role='admin' AND is_active=1")->fetchColumn();
```
На уровне списка это **не влияет напрямую** (бан/разбан/удаление делается через отдельные POST-эндпоинты, проверка там).

### Защита от самобана

В списке:
```php
foreach ($users as &$u) {
    $u['canBan'] = ($u['id'] !== $_SESSION['user_id']);
    $u['isSelf'] = ($u['id'] === $_SESSION['user_id']);
}
```
Кнопки бан/разбан не рендерятся для текущего админа.

## 6. UI: страница редактирования `admin/user-edit.php`

### Сценарий A: создание (`$_GET['id']` отсутствует)

**Поля формы** (метод POST, action = сам файл):

| Поле | Тип | Валидация | Обязательное |
|------|-----|-----------|--------------|
| Email | email | формат + уникальность | да |
| Имя | text | длина 2-100 | да |
| Фамилия | text | длина 2-100 | да |
| Телефон | tel | `^[+\d\s()-]{6,20}$` | нет |
| Роль | select (`user` / `agent`) | in_array | да |
| Временный пароль | text + кнопка «Сгенерировать» (JS) | длина ≥ 8 | да |

**Чекбоксы:**
- ☑ «Отправить пользователю письмо с логином и паролем» (по умолчанию вкл.)

**Кнопки:** «Создать» (submit), «Отмена» (→ `users.php`).

**Логика POST:**
1. CSRF-проверка.
2. Валидация полей.
3. Проверка уникальности email.
4. `password_hash($password, PASSWORD_BCRYPT)`.
5. `INSERT INTO users (email, password_hash, first_name, last_name, phone, role, is_active, email_verified_at) VALUES (?, ?, ?, ?, ?, ?, 1, NOW())`.
6. Если чекбокс «отправить письмо» — `Mailer::send($email, 'Добро пожаловать в Elsesser & Co.', $html)` с текстом, содержащим email и пароль (открытым текстом, с предупреждением «удалите это письмо после входа»).
7. Flash «Пользователь создан» → redirect → `users.php`.

### Сценарий B: редактирование (`?id=N`)

**Шапка:** «Редактирование пользователя #N — email».

**Поля формы** (те же, кроме пароля — пароль редактируется отдельно):

| Поле | Тип | Заметка |
|------|-----|---------|
| Email | email | readonly, можно снять readonly если не критично — НЕ снимаем, чтобы случайно не сломать OAuth-связку |
| Имя, Фамилия, Телефон | text | редактируемые |
| Роль | select (`user` / `agent` / `admin`) | для admin — но с проверкой «не последний админ» |
| Активен | checkbox (`is_active`) | disabled для текущего админа |
| Email подтверждён | info + кнопка «Подтвердить вручную» (если NULL) | кнопка скрыта если уже подтверждён |

**Блок «Безопасность»** (отдельная карточка):
- «Отправить ссылку на сброс пароля» (POST `?action=send-reset&id=N`).
- «Удалить пользователя» — красная кнопка, `confirm()` перед отправкой; disabled если это сам админ или последний админ.

**Информационный блок** (read-only):
- Зарегистрирован: `2026-04-12 14:33`
- Последний вход: `2026-05-30 09:12` (берём из `users.last_login_at` — если поля нет, см. §11)
- IP последнего входа: `…` (если есть)
- OAuth-провайдер: `vk` / `yandex` / `google` / `—` (если `oauth_provider IS NULL`)
- Неудачных попыток входа: `0`
- locked_until: `—` (или дата, если заблокирован)

**Логика POST (обновление):**
1. CSRF-проверка.
2. Загрузить пользователя. Если не найден — 404.
3. Валидация полей.
4. Проверка уникальности email (исключая текущего).
5. Проверка «не последний админ» (если меняется `role` с admin на другое, или `is_active` с 1 на 0, для админа).
6. Проверка «не сам себя» (если меняется `is_active` с 1 на 0 для текущего админа).
7. `UPDATE users SET first_name=?, last_name=?, phone=?, role=?, is_active=? WHERE id=?`.
8. Flash «Сохранено» → redirect → `users.php`.

**Логика POST `?action=send-reset`:**
1. CSRF-проверка.
2. Генерируем `bin2hex(random_bytes(32))` (64 hex).
3. `INSERT INTO password_resets (user_id, token_hash, expires_at) VALUES (?, SHA2(?, 256), NOW() + INTERVAL 1 HOUR)`.
4. `Mailer::send($email, 'Сброс пароля в Elsesser & Co.', $html)` с ссылкой `https://<host>/reset-password.php?token=<plain>`.
5. Flash «Ссылка отправлена» → stay on page.

**Логика POST `?action=delete`:**
1. CSRF-проверка.
2. Проверка «не сам себя» и «не последний админ».
3. `DELETE FROM users WHERE id=?` (FK-каскад сам уберёт favorites/inquiries/reviews/saved_searches).
4. Flash «Удалён» → redirect → `users.php`.

## 7. Защита от CSRF, ошибок, валидация

### CSRF

Все формы имеют `<input type="hidden" name="csrf_token" value="<?= csrf_token() ?>">`.
Все POST-обработчики начинаются с:
```php
if (!verifyCsrfToken($_POST['csrf_token'] ?? '')) {
    http_response_code(403);
    exit('Invalid CSRF token');
}
```

### Защита «не сам себя»

```php
$isSelf = ((int)$user['id'] === (int)$_SESSION['user_id']);
```

### Защита «не последний админ»

```php
function isLastActiveAdmin(PDO $pdo, int $userId): bool {
    $count = (int)$pdo->query(
        "SELECT COUNT(*) FROM users WHERE role='admin' AND is_active=1"
    )->fetchColumn();
    if ($count > 1) return false;
    // count == 1 — проверяем, что это именно этот пользователь
    $stmt = $pdo->prepare("SELECT role, is_active FROM users WHERE id = ?");
    $stmt->execute([$userId]);
    $u = $stmt->fetch();
    return $u && $u['role'] === 'admin' && (int)$u['is_active'] === 1;
}
```

### Валидация ввода

- **Email:** `filter_var($email, FILTER_VALIDATE_EMAIL)` + проверка уникальности `SELECT 1 FROM users WHERE email=? AND id != ?` (если редактирование).
- **Имя/Фамилия:** `trim`, `mb_strlen($v) >= 2 && mb_strlen($v) <= 100`.
- **Телефон:** допускаем `+`, цифры, пробелы, скобки, дефис; длина 6-20; пустая строка — ОК.
- **Роль:** `in_array($role, ['user','agent','admin'], true)` (whitelist).
- **Пароль (только при создании):** `strlen($pwd) >= 8 && preg_match('/[A-Za-z]/', $pwd) && preg_match('/\d/', $pwd)` — буквы + цифры, минимум 8.

### Ошибки

- Если `?id=N` не существует — `http_response_code(404)` с простой страницей «Пользователь не найден» и ссылкой «← Назад».
- Невалидная форма — рендерим ту же форму с заполненными значениями + `<div class="alert alert--error">` вверху.
- Действие, нарушающее «последний админ» / «сам себя» — `http_response_code(403)` + flash.
- Ошибка БД — `error_log(...)` + flash «Не удалось выполнить действие».

### Логирование

Действия админа **не логируются** в этой итерации (audit log отложен). Только `error_log` на исключениях.

## 8. Email-шаблоны

Создаём 2 шаблона (HTML-строки, лежат рядом с логикой отправки — не отдельные файлы, как в `Mailer::wrap()`).

### Шаблон A: «Добро пожаловать» (при создании пользователя)

```html
<h2>Здравствуйте, {first_name}!</h2>
<p>Для вас создан аккаунт в Elsesser & Co.</p>
<p><b>Email для входа:</b> {email}</p>
<p><b>Временный пароль:</b> {password}</p>
<p style="color:#dc2626">
  ⚠ Рекомендуем сменить пароль после первого входа в личном кабинете.
  Удалите это письмо после использования.
</p>
<p>Ссылка для входа: <a href="https://<?= $_SERVER['HTTP_HOST'] ?>/login.php">войти</a>.</p>
```

### Шаблон B: «Сброс пароля» (при отправке ссылки)

```html
<h2>Здравствуйте, {first_name}!</h2>
<p>По вашему запросу создана ссылка для сброса пароля.</p>
<p><a href="https://<?= $_SERVER['HTTP_HOST'] ?>/reset-password.php?token={token}">
  Сбросить пароль
</a></p>
<p>Ссылка действительна 1 час. Если вы не запрашивали сброс — просто проигнорируйте это письмо.</p>
```

Оба шаблона — простые HTML-строки, в которые подставляются `{first_name}`, `{email}`, `{password}`, `{token}`. `Mailer::wrap()` обернёт их в фирменный контейнер.

## 11. Открытые вопросы (требуют уточнения при планировании)

1. **Поле `last_login_at` / `last_login_ip` в `users`** — нет ли его в текущей схеме? Если нет, информационный блок в `user-edit.php` покажет «—» для этих полей. **Решение:** оставляем «—» (без миграции в этой итерации).

2. **Используется ли уже где-то `csrf_token()` / `verifyCsrfToken()`** — посмотреть в `includes/auth/` при планировании. Если нет — реализуем простой helper в `includes/auth/csrf.php`.

3. **Flash-сообщения** — посмотреть, есть ли уже helper. Если нет — реализуем простую пару `flashSet()` / `flashGet()` (2 функции в `includes/flash.php`).

4. **Где в `admin.css` определены таблицы/бейджи** — посмотреть при планировании, чтобы не дублировать.

## 12. Что НЕ делаем (явный out-of-scope)

- Audit log действий админа (`admin_actions` таблица).

- Экспорт пользователей в CSV.
- Email-уведомления пользователю о бане / подтверждении email.

- Рассылки (тема для следующего спека).

- Возможность админу самому сбрасывать пароль пользователю (создавать новый пароль).
